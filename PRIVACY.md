# Privacy and network boundaries

LAC Core has no telemetry or product analytics. Its default Ollama endpoint is the loopback service at `http://localhost:11434`, and the desktop web server binds to `127.0.0.1` by default. Source and headless operators can override the server's `--host`; binding it to a non-loopback address makes the interface and its unauthenticated state-changing API reachable from that network. Any client that can connect may invoke exposed endpoints, including model-management actions. Never bind LAC directly to the public internet; use a trusted firewall and an authenticated reverse proxy if remote access is required.

## When data can leave the machine

- **Configured Ollama endpoint.** Prompts, chat history supplied as context, and model output are sent to the configured Ollama endpoint. That endpoint defaults to this machine but may be remote. If you configure a remote endpoint, those values leave the machine and are handled by that endpoint's operator.
- **Configured AI providers.** A project can be configured to use the built-in OpenAI-compatible or Anthropic provider, or a provider supplied by a plugin. LAC sends that provider the selected model, system instructions, prompts, supplied chat history, tool definitions and tool results needed for the request. The provider's own terms and retention policy apply. Provider API keys can come from environment variables or project configuration; prefer environment variables and never commit a key.
- **Agent web search.** When the Explore agent invokes web search, LAC sends the exact search query to DuckDuckGo and returns result titles. DuckDuckGo receives the query and normal network metadata. Do not put secrets, personal data, or confidential client information in a search prompt.
- **Configured MCP servers.** When an agent invokes a tool from a configured Model Context Protocol (MCP) server, LAC sends that server the selected tool name and arguments. HTTP and SSE MCP servers may be remote; a local stdio server is separate software and can make its own network requests. Tool arguments can contain prompt-derived or project-derived data, so configure only servers you trust.
- **Ollama library metadata.** On server start, LAC reads its public model-library cache and may automatically refresh stale or missing metadata from `https://ollama.com/library`. This request sends normal network metadata but no prompts or chat history. The cache is considered stale after one hour.
- **GitHub.** Source/headless mode checks GitHub Releases when the server starts. The frozen Windows desktop skips that automatic startup check; its update-check API contacts GitHub only if invoked. Opening project or release links also contacts GitHub through your browser. GitHub receives normal network metadata for these requests, but LAC does not send prompts or chat history for update checks.
- **Hugging Face and model sources.** Browsing model metadata or importing a model can contact Hugging Face or another explicitly selected model source. When a Hugging Face token is configured, Local Pro attaches it to its Hugging Face API requests, including public-repository metadata requests. The token is currently stored as plaintext JSON at `~/.model-hub/hf_token.json` and relies on the user account and filesystem ACLs for protection. Use a fine-grained read-only token with the minimum repository access needed. Secure token storage remains a Local Pro launch control; never paste tokens into logs or issue reports.
- **Pro activation.** If Local Pro is installed, activation sends the license key, the machine hostname as its seat label, and a locally generated device ID to the configured Pro activation service. Periodic entitlement validation sends the key and device-binding identifiers at most once every three days; a valid cached receipt has up to 14 days of offline grace when the service is unavailable. Model execution and tuning use the configured Ollama endpoint, which is local by default but may be remote. Free LAC installations do not contain Local Pro code.
- **Account and Cloud surfaces.** The desktop includes a fail-closed account bridge, but the production Cloud origin is deliberately unconfigured in unapproved builds. A source operator can deliberately configure an approved staging origin; signing in then sends OAuth session and account requests to that service. Cloud execution remains disabled until the launch gate approves the service, providers, metering, retention, deletion, security, and operational evidence.
- **Links you choose to open.** The project website and application link to external sites such as GitHub, Ollama, Hugging Face, and the Tally waitlist. Those sites apply their own privacy terms after you open them.

## Local data and removal

LAC stores settings, model metadata, caches, and diagnostic results under the current user's local profile. Its `~/.model-hub/cookbook.db` SQLite database stores registered project metadata and root paths, session names, models, system prompts, chat messages, context, tool and session event payloads, and the full old and new content of staged files. This data is not encrypted by LAC; the protections on your user account and filesystem apply.

Uninstalling LAC does not delete `~/.model-hub`, Ollama models, provider-side records, or data owned by external services. After uninstalling, and after preserving anything you still need, manually remove `~/.model-hub` to delete LAC's retained local profile data. Use each external service's controls for data it received.

Before sharing logs or diagnostics, review them for file paths, model names, prompts, account details, and other information you do not want to disclose. Never include passwords, API keys, license keys, or tokens in a public issue.

This document describes the repository-controlled behavior of the current source release candidate. A future hosted service will require its own approved privacy terms before Cloud execution can be enabled.
